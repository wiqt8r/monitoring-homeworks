# Домашнее задание к занятию "13.Системы мониторинга"

## Обязательные задания

> 1. Вас пригласили настроить мониторинг на проект. На онбординге вам рассказали, что проект представляет из себя 
платформу для вычислений с выдачей текстовых отчетов, которые сохраняются на диск. Взаимодействие с платформой 
осуществляется по протоколу http. Также вам отметили, что вычисления загружают ЦПУ. Какой минимальный набор метрик вы
выведите в мониторинг и почему?

Исходя из описания проекта (HTTP-платформа, вычисления нагружают CPU, результаты сохраняются на диск), минимальный набор метрик:
* Инфраструктурные метрики:
 * CPU usage, load average - вычисления нагружают ЦПУ, поэтому важно видеть утилизацию и перегрузки.
 * RAM usage - взможны утечки памяти или нехватка RAM при обработке отчетов.
 * Disk usage - отчеты сохраняются на диск,он может переполняться.
 * Disk I/O - может быть узким местом при записи отчетов.
 * Inodes usage - при большом количестве файлов диск может закончиться по inodes.
* Метрики сервиса:
 * HTTP requests per second - для понимания нагрузки на платформу.
 * HTTP response codes (4xx/5xx) - для оценки корректности работы сервиса.
 * HTTP latency (p50/p95/p99) - чтобы отразить производительность с точки зрения пользователя.
 * Очереди и время выполнения задач - чтобы показать, успевает ли система обрабатывать вычисления.

> 2. Менеджер продукта посмотрев на ваши метрики сказал, что ему непонятно что такое RAM/inodes/CPUla. Также он сказал, 
что хочет понимать, насколько мы выполняем свои обязанности перед клиентами и какое качество обслуживания. Что вы 
можете ему предложить?

Менеджеру не нужны технические метрики, ему нужны бизнес-ориентированные показатели.
Что можно ему предложить:
1. SLI / SLO / SLA
* Доступность сервиса (например, 99% успешных HTTP-запросов)
* Время ответа (наприме, 95% запросов обрабатываются быстрее 500 мс)
* Процент успешных расчетов (отношение успешно завершенных вычислений к общему числу)
2. Метрики с ориентацией на пользователей:
* Количество успешно сформированных отчетов
* Среднее время генерации отчета
* Количество ошибок, влияющих на пользователя
3. Визуализация в понятных терминах:
* Вместо технических терминов CPU load и RAM usage можно предложить RAG-статусы, типа
 * Зеленый - Сервис работает стабильно
 * Желтый - Наблюдается замедление или иная проблема
 * Красный - Критическая проблема / нарушение SLA

Итог: менеджер видит качество обслуживания клиентов, а не технику "под капотом" системы.

> 3. Вашей DevOps команде в этом году не выделили финансирование на построение системы сбора логов. Разработчики в свою 
очередь хотят видеть все ошибки, которые выдают их приложения. Какое решение вы можете предпринять в этой ситуации, 
чтобы разработчики получали ошибки приложения?

Возможные решения без отдельной системы логирования:
1. Логи в stdout/stderr + systemd/journald
Разработчики пишут ошибки в stderr, используем journalctl для поиска и фильтрации ошибок
2. Настроить логирование только ERROR в приложениях, не писать уровен DEBUG/INFO/WARN. Ротировать логи через logrotate. Использовать grep/awk/tail и текстовые редакторы для анализа.
3. Настроить алерты по ключевым ошибкам. Парсить логи самописными скриптами в cron. Отправлять уведомления по email/slack/telegram

Это не будет полноценный observability-стек, но разработчики будут видеть ошибки и реагировать на них в разумные сроки.

> 4. Вы, как опытный SRE, сделали мониторинг, куда вывели отображения выполнения SLA=99% по http кодам ответов. 
Вычисляете этот параметр по следующей формуле: summ_2xx_requests/summ_all_requests. Данный параметр не поднимается выше 
70%, но при этом в вашей системе нет кодов ответа 5xx и 4xx. Где у вас ошибка?

SLA считается некорректно из-за неправильного знаменателя в формуле. В `summ_all_requests` попадают все коды, а не только двухсотые. Это могут быть 1xx, 3xx, таймауты, запросы без ответа, connection reset. Это и есть 30%. Скорее всего, на сайте настроены заголовки upgrade или редиректы.
Надо либо учитывать только завершенные запросы, либо явно определить, что считается успешным SLI.

Можно использовать такую формулу: `2xx / (2xx + 4xx + 5xx)`

Или такую: `(successful_requests / total_finished_requests)`

> 5. Опишите основные плюсы и минусы pull и push систем мониторинга.

**Pull-модель**

Плюсы:

* Централизованный контроль
* Легче дебажить
* Нет доверия к агентам (сервер сам забирает данные)
* Хорошо масштабируется

Минусы:

* Проблемы с динамической инфраструктурой (NAT, firewalls)
* Нужно открывать эндпоинты

**Push-модель**

Плюсы:
* Работает за NAT
* Подходит для batch-job и short-lived задач
* Проще для клиентов

Минусы:
* Сложнее контролировать целостность данных
* Риск перегрузки сервера
* Сложнее обнаружить пропавшие хосты
